## 1. 产品概述

### 1.1 产品定位

飞书群消息整理工具是一款命令行自动化工具，用于从飞书群聊中提取分享的网络链接，自动解析文章信息，并结构化写入飞书电子表格。

### 1.2 目标用户

信息收集者——需要定期整理飞书群中分享的文章链接，建立结构化的文章索引库。

### 1.3 核心价值

- 将群聊中零散的链接分享转化为结构化的文章数据库
- 自动解析文章元信息（标题、日期、来源），减少人工整理成本
- 增量处理 + 重复检测，保证数据不遗漏、不重复
- 处理完成后可自动撤回消息，保持群聊整洁

## 2. 核心业务流程

用户旅程分为四个阶段：**准备 → 采集 → 入库 → 清理**。

```mermaid
flowchart TD
    A[开始] --> B{Token 有效?}
    B -->|是| D[读取群聊消息]
    B -->|否| C{refresh_token 有效?}
    C -->|是| C1[自动刷新 Token]
    C1 --> D
    C -->|否| C2[提示用户运行 auth.py]
    C2 --> END1[退出]

    D --> E{增量 or 全量?}
    E -->|默认| F[只取新消息]
    E -->|--all / --reset| G[取全部历史消息]
    F --> H[提取链接]
    G --> H

    H --> I{消息含链接?}
    I -->|否| I1[计入无链接统计]
    I -->|是| J[识别来源 + URL截断]

    J --> K[解析文章: 标题/日期]
    K --> L{解析成功?}
    L -->|是| M[组装完整记录]
    L -->|部分失败| M1[记录异常信息] --> M

    M --> N[读取表格已有URL]
    N --> O{URL重复?}
    O -->|是| O1[标记"重复"] --> P
    O -->|否| P[检查Pin状态]
    P --> P1{是Pin消息?}
    P1 -->|是| P2[标记"Pin"] --> Q
    P1 -->|否| Q[写入飞书表格]

    Q --> R{--recall 参数?}
    R -->|是| S[撤回已处理消息]
    R -->|否| T[更新处理时间记录]
    S --> T
    T --> END2[完成]
```

## 3. 用户故事

### 阶段一：准备

#### US-01 授权与 Token 刷新

**作为** 用户，**我希望** 程序自动管理 Token 生命周期，**以便** 日常使用时不需要频繁手动授权。

**前置条件**：已在飞书开放平台创建应用并配置权限。

**正常路径**：
- 启动时检查 user_access_token 有效性
- 过期则用 refresh_token 自动刷新
- 新 token 写回 config.yaml

**异常路径**：
- refresh_token 过期（>30天）或刷新 API 调用失败 → 提示用户运行 `python src/auth.py` 重新授权 → 程序退出

**边界场景**：
- 首次使用无 token → 直接提示运行 auth.py 并退出

**验收标准**：
| 编号 | 条目 |
|------|------|
| AC-1 | token 有效时程序正常执行，无额外提示 |
| AC-2 | user_access_token 过期、refresh_token 有效时，自动刷新成功并继续执行 |
| AC-3 | refresh_token 过期时，输出明确的重新授权提示并退出 |
| AC-4 | 网络异常导致刷新失败时，同样提示重新授权并退出 |
| AC-5 | 首次使用无 token 时，提示运行 auth.py 并退出 |

### 阶段二：采集

#### US-02 链接提取

**作为** 信息收集者，**我希望** 自动从飞书群聊消息中提取所有网络链接，**以便** 不遗漏任何分享的文章。

**前置条件**：Token 有效（US-01 已通过）。

**业务规则**：
- 处理文本消息和富文本消息（post），其他类型消息（卡片、图片等）跳过
- 提取消息中所有 HTTP/HTTPS 链接
- 一条消息含多个链接时，每个链接各自作为一条记录，共享同一个消息序号

**无链接消息处理**：
- 仅计入统计数量，不做其他处理
- `--list-nolink` 可查看详情，不影响主流程

**验收标准**：
| 编号 | 条目 |
|------|------|
| AC-1 | 能正确提取文本消息和富文本消息中的链接 |
| AC-2 | 一条消息含多个链接时，生成多条记录且消息序号相同 |
| AC-3 | 非文本/富文本消息（卡片、图片等）被跳过，不报错 |
| AC-4 | 无链接消息正确计入统计 |
| AC-5 | `--list-nolink` 模式下只展示无链接消息列表，不执行后续解析和写入 |

#### US-03 文章解析

**作为** 信息收集者，**我希望** 自动解析链接对应的文章标题、日期和来源，**以便** 快速了解文章基本信息。

**前置条件**：链接已提取（US-02）。

**业务规则**：
- 基于 URL 特征匹配 site_rules 识别来源平台
- 匹配到截断规则的 URL，先截断（去除 ? 及之后参数）再请求
- 各平台使用专用解析方法提取标题和日期，无专用方法时走通用 fallback
- 日期优先取最新更新日期，取不到时 fallback 到创建日期

**异常处理**：
- 标题、日期、来源三项独立解析，某项失败不影响其他项
- 部分失败时在"摘录异常信息"列记录具体缺失项（如"未提取标题"、"未提取日期"）
- 网站无法访问（超时、403、需登录等）→ 记录异常信息，继续处理下一条

**验收标准**：
| 编号 | 条目 |
|------|------|
| AC-1 | 微信公众号文章能解析出标题、日期、公众号名称 |
| AC-2 | 飞书文档能解析出标题和更新日期 |
| AC-3 | 头条视频/文章能解析出标题和日期 |
| AC-4 | 需截断的 URL 在请求前已去除追踪参数 |
| AC-5 | 标题提取失败、日期成功时，记录"未提取标题"，日期正常填写 |
| AC-6 | 网站无法访问时记录异常，不中断后续处理 |
| AC-7 | 未匹配任何 site_rule 的链接，使用通用方法解析 |

### 阶段三：入库

#### US-04 表格写入

**作为** 信息收集者，**我希望** 解析结果自动写入飞书电子表格，**以便** 在统一的地方管理所有文章记录。

**前置条件**：文章已解析（US-03），重复检测和 Pin 标记已完成（US-06、US-07）。

**业务规则**：
- 批量一次性写入飞书电子表格
- 写入顺序：按消息时间排序，同一消息内多个链接按出现顺序
- 写入字段：消息序号、文章标题、发表日期、星期、链接URL、摘录异常信息、来源、备注、是否重复、标记

**异常处理**：
- 飞书 API 报错（限流、网络中断等）→ 自动重试，最多3次
- 3次重试均失败 → 报错退出

**验收标准**：
| 编号 | 条目 |
|------|------|
| AC-1 | 所有记录批量写入表格末尾 |
| AC-2 | 写入顺序与消息时间及链接出现顺序一致 |
| AC-3 | 10个字段全部正确填写 |
| AC-4 | API 偶发失败时重试成功，不影响最终结果 |
| AC-5 | 3次重试均失败时，输出明确错误信息 |

#### US-05 增量处理

**作为** 信息收集者，**我希望** 程序默认只处理新消息，**以便** 避免重复工作。

**业务规则**：
- 通过 config.yaml 中的 last_processed_time 记录上次处理时间点
- 默认模式：只获取 last_processed_time 之后的新消息，处理完后更新时间
- `--all`：获取全部历史消息，处理完后更新时间
- `--reset`：先将 last_processed_time 清零，再获取全部历史消息，处理完后更新时间
- `--start N`/`--end N`：限制处理范围（按链接序号），处理完后更新时间

**各模式区别**：
- 默认 vs `--all`：处理范围不同（增量 vs 全量），都更新时间
- `--all` vs `--reset`：`--reset` 会先清零时间再处理，`--all` 直接处理全量

**验收标准**：
| 编号 | 条目 |
|------|------|
| AC-1 | 默认运行只处理新消息 |
| AC-2 | 连续两次默认运行，第二次无新数据时不产生新记录 |
| AC-3 | `--all` 处理全部历史消息并更新时间记录 |
| AC-4 | `--reset` 清零后处理全部消息并更新时间记录 |
| AC-5 | `--start`/`--end` 正确限制处理范围 |

#### US-06 重复检测

**作为** 信息收集者，**我希望** 自动检测重复的文章链接，**以便** 知道哪些文章已经收录过。

**前置条件**：链接已提取并完成 URL 截断（US-02、US-03）。

**业务规则**：
- 比对使用截断后的 URL
- 先读取表格中已有的 URL 列表作为基准
- 新链接与已有记录比对，匹配则标记"重复"
- 同一批次内出现相同 URL，后出现的也标记"重复"
- 重复记录不跳过，仍然正常写入表格

**验收标准**：
| 编号 | 条目 |
|------|------|
| AC-1 | 与表格已有记录 URL 相同的链接被标记"重复" |
| AC-2 | 同一批次内重复的链接，后出现的被标记"重复" |
| AC-3 | 重复记录正常写入表格，不被跳过 |
| AC-4 | URL 截断后再比对（带参数和不带参数的同一链接视为重复） |
| AC-5 | 非重复记录"是否重复"列为空 |

#### US-07 Pin 标记

**作为** 信息收集者，**我希望** 自动识别被 Pin 的消息，**以便** 在表格中区分重要文章。

**前置条件**：Token 有效（US-01）。

**业务规则**：
- 处理开始时通过飞书 API 一次性获取群聊 Pin 消息列表
- 将提取的链接与 Pin 列表匹配，命中则在"标记"列写入"Pin"
- 一条 Pin 消息含多个链接时，所有链接记录都标记"Pin"
- 只做正向标记，不回溯清理（消息取消 Pin 后，已写入的标记不会被删除）

**验收标准**：
| 编号 | 条目 |
|------|------|
| AC-1 | Pin 消息对应的链接记录"标记"列为"Pin" |
| AC-2 | 一条 Pin 消息的多个链接全部标记"Pin" |
| AC-3 | 非 Pin 消息的"标记"列为空 |
| AC-4 | 消息取消 Pin 后，已写入的"Pin"标记不受影响 |

### 阶段四：清理

#### US-08 消息撤回

**作为** 群管理员，**我希望** 已转移到表格的消息能从群聊中撤回，**以便** 保持群聊整洁。

**前置条件**：Token 有效（US-01），授权用户为群管理员或群主。

**两种使用方式**：
- 主程序 `--recall`：处理完成后自动撤回本次处理过的所有含链接消息；可叠加 `--confirm-each` 逐条确认
- 独立脚本 `recall_messages.py`：按序号手动指定撤回

**业务规则**：
- 倒序撤回（从编号最大的开始），避免序号混乱
- 序号格式支持：单个（1）、多个（1,3,5）、范围（1-5）、混合（1,3-5,7）
- `--confirm-each` 模式：逐条提示，y=撤回、n=跳过、a=确认剩余全部、q=退出
- `--dry-run`：只展示将要撤回的消息列表，不执行任何操作

**异常处理**：
- 撤回失败（权限不足、消息已被撤回等）→ 记录失败原因，继续撤回下一条

**已知限制**：撤回后群内会显示"XXX撤回了一条消息"提示，无法静默删除。

**验收标准**：
| 编号 | 条目 |
|------|------|
| AC-1 | `--recall` 自动撤回本次处理的含链接消息 |
| AC-2 | 撤回顺序为倒序 |
| AC-3 | `--confirm-each` 模式下 y/n/a/q 四个选项行为正确 |
| AC-4 | `--dry-run` 只展示不执行 |
| AC-5 | 单条撤回失败不中断，记录失败后继续下一条 |
| AC-6 | 混合序号格式（如 1,3-5,7）正确解析 |

## 4. 非功能需求

### 4.1 性能
- 请求频率控制，避免触发飞书 API 限流
- 分页获取消息，支持大量历史消息处理

### 4.2 安全
- config.yaml 包含敏感信息，不提交到版本控制
- Token 自动刷新减少明文凭证暴露
- 撤回操作需管理员权限

### 4.3 可扩展性
- 规则驱动的网站识别架构，新增网站只需添加 site_rules 条目
- 各网站独立解析方法，互不影响

### 4.4 兼容性
- Python 3.7+
- Windows / Linux / macOS
- 飞书开放平台 API v2

## 5. 已知限制

1. **飞书撤回提示**：撤回消息后群内会显示"XXX撤回了一条消息"，无法静默删除
2. **Flag 标记不可识别**：消息 Flag 是客户端本地行为，服务端无 API
3. **Token 时效**：user_access_token 约2小时有效，refresh_token 30天有效
4. **飞书文档权限**：部分文档因权限限制只能获取创建日期
5. **登录限制**：小红书、微博等平台内容可能需要登录才能访问
6. **微博解析受限**：App-微博通常只能识别来源，无法提取标题和日期

### 7.3 命令行参数
