# 消息撤回常见问题

## Q1: 撤回消息后，为什么消息序号还是连续的？

### 答案：消息序号是动态分配的，不是固定的

**工作机制：**

1. **第一次运行 `--list`**
   ```
   获取到 50 条消息
   [消息 1] 内容A
   [消息 2] 内容B
   [消息 3] 内容C
   ...
   [消息 50] 内容Z
   ```

2. **撤回消息 1, 3, 5**
   ```bash
   python src/archive/recall_messages.py --indices 1,3,5
   ```
   - 消息A、C、E 从飞书服务器永久撤回
   - 撤回成功后会显示验证信息

3. **第二次运行 `--list`**
   ```
   获取到 47 条消息（50 - 3 = 47）
   [消息 1] 内容B  ← 原来的消息2
   [消息 2] 内容D  ← 原来的消息4
   [消息 3] 内容F  ← 原来的消息6
   ...
   [消息 47] 内容Z
   ```

**关键点：**
- ✅ 消息确实从服务器永久撤回了
- ✅ 剩余消息会被重新编号为 1, 2, 3...
- ✅ 消息序号只在当前运行中有效

## Q2: 如何验证消息是否真的被撤回？

### 方法1：查看消息总数变化

撤回完成后，程序会自动显示验证信息：

```
============================================================
撤回完成
============================================================
成功: 3 条
失败: 0 条
总计: 3 条

正在验证撤回结果...
✓ 撤回前消息总数: 50
✓ 撤回后消息总数: 47
✓ 实际减少: 3 条
✓ 验证成功：消息已从服务器永久撤回
```

### 方法2：使用调试模式

```bash
python src/archive/recall_messages.py --indices 1 --debug
```

调试模式会显示详细的API响应信息，包括：
- API返回的状态码
- 错误信息（如果有）
- 撤回是否成功

### 方法3：对比消息内容

1. 撤回前记录要撤回的消息内容
2. 撤回后再次运行 `--list`
3. 确认该内容的消息不再出现

## Q3: 为什么有些消息撤回失败？

### 常见原因：

1. **权限不足**
   ```
   撤回失败 (code: 1301): no permission
   ```
   - 需要是群管理员
   - 或者只能撤回自己发送的消息

2. **消息已被撤回**
   ```
   撤回失败 (code: 230002): message not found
   ```
   - 消息可能已经被其他人撤回
   - 或者消息ID无效

3. **Token过期**
   ```
   撤回失败 (code: 99991663): token invalid
   ```
   - 运行 `python src/auth.py` 重新授权

## Q4: 如何确保撤回的是正确的消息？

### 推荐流程：

1. **列出所有消息**
   ```bash
   python src/archive/recall_messages.py --list
   ```
   记录要撤回的消息序号和内容

2. **试运行**
   ```bash
   python src/archive/recall_messages.py --indices 1,3,5 --dry-run
   ```
   确认将要撤回的消息内容

3. **逐条确认撤回**
   ```bash
   python src/archive/recall_messages.py --indices 1,3,5 --confirm-each
   ```
   每条消息都会显示内容并要求确认

4. **验证结果**
   撤回完成后，程序会自动验证消息总数变化

## Q5: 消息序号会固定吗？

### 答案：不会

消息序号的特点：
- ❌ **不是固定的**：每次运行都会重新分配
- ❌ **不是消息ID**：消息ID是飞书服务器分配的唯一标识
- ✅ **只在当前运行中有效**：用于方便用户选择要撤回的消息
- ✅ **按时间顺序编号**：从最早的消息开始编号为1

**为什么这样设计？**
- 简化用户操作：用户只需记住简单的数字（1, 2, 3...）
- 避免混淆：撤回后重新编号，用户看到的始终是连续的序号
- 灵活性：每次运行都是全新的视图

## Q6: 如何撤回特定内容的消息？

### 方法：先列出，再根据内容选择序号

```bash
# 1. 列出所有消息
python src/archive/recall_messages.py --list > messages.txt

# 2. 查看 messages.txt，找到包含特定内容的消息序号
# 例如：找到所有包含"广告"的消息是 5, 12, 18

# 3. 撤回这些消息
python src/archive/recall_messages.py --indices 5,12,18 --confirm-each
```

## Q7: 撤回操作可以撤销吗？

### 答案：不可以

- ❌ 撤回操作是**永久性**的
- ❌ 无法从飞书服务器恢复已撤回的消息
- ✅ 建议使用 `--dry-run` 先预览
- ✅ 建议使用 `--confirm-each` 逐条确认

## Q8: 批量撤回大量消息的最佳实践

### 推荐步骤：

1. **先小范围测试**
   ```bash
   # 先撤回1条测试
   python src/archive/recall_messages.py --indices 1 --debug
   ```

2. **使用范围参数**
   ```bash
   # 撤回前10条
   python src/archive/recall_messages.py --indices 1-10 --dry-run
   ```

3. **逐条确认重要操作**
   ```bash
   # 逐条确认，可以跳过重要消息
   python src/archive/recall_messages.py --indices 1-50 --confirm-each
   ```

4. **验证结果**
   - 查看撤回完成后的验证信息
   - 再次运行 `--list` 确认

## 技术说明

### 消息序号 vs 消息ID

| 特性 | 消息序号 | 消息ID |
|------|---------|--------|
| 格式 | 简单数字（1, 2, 3...） | 复杂字符串（om_xxx） |
| 唯一性 | 仅在当前运行中唯一 | 全局唯一 |
| 持久性 | 每次运行重新分配 | 永久不变 |
| 用途 | 用户交互 | API调用 |
| 可读性 | 高 | 低 |

### 撤回流程

```
用户输入序号 → 查找对应消息ID → 调用撤回API → 验证结果
    ↓              ↓                ↓              ↓
  1,3,5      om_xxx, om_yyy    DELETE请求    对比消息总数
```

## 总结

- ✅ 消息撤回是永久的，会从服务器移除
- ✅ 消息序号是动态的，每次运行重新分配
- ✅ 使用验证功能确认撤回成功
- ✅ 使用 `--dry-run` 和 `--confirm-each` 避免误删
- ✅ 使用 `--debug` 查看详细的API响应
