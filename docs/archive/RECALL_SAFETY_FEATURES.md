# 撤回消息安全特性演示

## 功能 1：倒序撤回

### 为什么需要倒序撤回？

当撤回多条消息时，如果按正序撤回（1 → 2 → 3），会出现以下问题：

**问题场景：**
```
原始消息列表：
[1] 消息A
[2] 消息B
[3] 消息C
[4] 消息D
[5] 消息E

要撤回：1, 3, 5
```

**正序撤回的问题：**
```
1. 撤回消息1（消息A）
   → 剩余：[1]消息B, [2]消息C, [3]消息D, [4]消息E
   → 原来的消息2变成了消息1！

2. 撤回消息3（原来的消息C）
   → 但现在消息3是原来的消息D！
   → 误撤回了错误的消息！
```

**倒序撤回的优势：**
```
1. 撤回消息5（消息E）
   → 剩余：[1]消息A, [2]消息B, [3]消息C, [4]消息D
   → 前面的消息序号不受影响

2. 撤回消息3（消息C）
   → 剩余：[1]消息A, [2]消息B, [3]消息D
   → 前面的消息序号仍然不受影响

3. 撤回消息1（消息A）
   → 剩余：[1]消息B, [2]消息D
   → 正确撤回了目标消息
```

### 实现方式

程序会自动将要撤回的消息按序号倒序排列：

```python
# 从编号最靠后的消息开始撤回（倒序），避免序号混乱
messages_to_delete.sort(key=lambda x: x['index'], reverse=True)
```

### 使用示例

```bash
python src/archive/recall_messages.py --indices 1,3,5,7,9
```

**输出：**
```

✓ 将按倒序撤回（从编号最大的消息开始），避免序号混乱

开始撤回消息...

[1/5] 消息 9
  内容: 这是第9条消息
  ✓ 撤回成功

[2/5] 消息 7
  内容: 这是第7条消息
  ✓ 撤回成功

[3/5] 消息 5
  内容: 这是第5条消息
  ✓ 撤回成功

[4/5] 消息 3
  内容: 这是第3条消息
  ✓ 撤回成功

[5/5] 消息 1
  内容: 这是第1条消息
  ✓ 撤回成功
```

## 功能 2：逐条确认

### 为什么需要逐条确认？

批量撤回时，可能需要根据消息内容决定是否撤回某条消息。逐条确认模式允许你：
- 查看每条消息的完整内容
- 选择性跳过某些消息
- 随时退出撤回操作

### 使用方式

添加 `--confirm-each` 参数：

```bash
python src/archive/recall_messages.py --indices 1-10 --confirm-each
```

### 交互流程

```
============================================================
⚠️  警告：撤回操作不可恢复！
============================================================

✓ 已启用逐条确认模式
  将在撤回每条消息前单独确认

开始撤回消息...

[1/10] 消息 10
  内容: 这是一条测试消息，包含重要信息
  确认撤回此消息？(y=撤回, n=跳过, q=退出): n
  ⊘ 已跳过

[2/10] 消息 9
  内容: 这是一条广告消息
  确认撤回此消息？(y=撤回, n=跳过, q=退出): y
  ✓ 撤回成功

[3/10] 消息 8
  内容: 这是一条重要通知
  确认撤回此消息？(y=撤回, n=跳过, q=退出): q
  ✗ 用户选择退出，停止撤回

============================================================
撤回完成
============================================================
成功: 1 条
失败: 0 条
跳过: 1 条
总计: 10 条
```

### 确认选项

| 输入 | 操作 | 说明 |
|------|------|------|
| `y` | 撤回此消息 | 确认撤回当前消息 |
| `n` | 跳过此消息 | 保留当前消息，继续处理下一条 |
| `q` | 退出撤回 | 停止撤回操作，未处理的消息保留 |

### 消息内容显示

- 显示完整的消息内容（最多 200 字符）
- 超过 200 字符会自动截断并显示 "..."
- 包含消息序号和消息 ID

## 组合使用

### 场景 1：先试运行，再逐条确认

```bash
# 1. 先查看将要撤回的消息
python src/archive/recall_messages.py --indices 1-20 --dry-run

# 2. 确认后，逐条撤回
python src/archive/recall_messages.py --indices 1-20 --confirm-each
```

### 场景 2：处理所有历史消息，逐条确认

```bash
# 处理所有历史消息，并逐条确认
python src/archive/recall_messages.py --all --indices 1-50 --confirm-each
```

### 场景 3：批量撤回但保留部分消息

```bash
# 撤回 1-100 条消息，但可以选择性跳过某些重要消息
python src/archive/recall_messages.py --indices 1-100 --confirm-each
```

## 安全建议

### 推荐的操作流程

1. **列出所有消息**
   ```bash
   python src/archive/recall_messages.py --list
   ```
   查看所有消息及其序号，确定要撤回的范围。

2. **试运行**
   ```bash
   python src/archive/recall_messages.py --indices 1-10 --dry-run
   ```
   查看将要撤回的消息列表，确认无误。

3. **逐条确认撤回**
   ```bash
   python src/archive/recall_messages.py --indices 1-10 --confirm-each
   ```
   逐条查看并确认，可以选择性跳过某些消息。

### 何时使用逐条确认

**推荐使用：**
- ✅ 撤回大量消息（10 条以上）
- ✅ 不确定具体要撤回哪些消息
- ✅ 需要根据内容判断是否撤回
- ✅ 进行重要操作时

**可以不使用：**
- ⭕ 只撤回 1-2 条消息
- ⭕ 已经通过 --dry-run 确认过
- ⭕ 明确知道要撤回的消息

## 技术实现

### 倒序撤回实现

```python
# 从编号最靠后的消息开始撤回（倒序），避免序号混乱
messages_to_delete.sort(key=lambda x: x['index'], reverse=True)
print(f"✓ 将按倒序撤回（从编号最大的消息开始），避免序号混乱")
```

### 逐条确认实现

```python
# 逐条确认模式
if args.confirm_each:
    # 显示消息内容
    text = msg_info['text']
    if len(text) > 200:
        text = text[:200] + "..."
    print(f"  内容: {text}")

    confirm = input(f"  确认撤回此消息？(y=撤回, n=跳过, q=退出): ").lower()

    if confirm == 'q':
        print(f"\n✗ 用户选择退出，停止撤回")
        break
    elif confirm != 'y':
        print(f"  ⊘ 已跳过")

        continue
```

## 常见问题

### Q: 倒序撤回会影响撤回结果吗？

A: 不会。倒序撤回只是改变了撤回的顺序，最终撤回的消息是一样的。但倒序撤回可以避免序号混乱导致的误删。

### Q: 逐条确认时可以修改撤回范围吗？

A: 不能直接修改，但可以通过 `no` 跳过不想撤回的消息，或通过 `quit` 提前退出。

### Q: 如果撤回过程中出错怎么办？

A: 程序会继续处理下一条消息，不会因为一条消息撤回失败而停止。最后会显示成功、失败和跳过的统计。

### Q: 逐条确认模式下，跳过的消息会被记录吗？

A: 会在最终统计中显示跳过的数量，但不会记录到配置文件中。

## 总结

两个安全特性的设计目标：

1. **倒序撤回**：防止误删，确保撤回的是正确的消息
2. **逐条确认**：提供灵活性，允许根据内容选择性撤回

这两个特性可以单独使用，也可以组合使用，为撤回操作提供多层保护。
