# AGENTS.md

本文件是本仓库的项目级 agent 工作规范。任何自动化编码 agent、协作式 AI 助手或后续 session 在修改本项目时，应优先阅读并遵守本文档。

目标不是增加流程负担，而是让每次修改都可接力、可验证、可复盘，避免同一类问题在不同 session 中反复丢失上下文。

## 1. 项目上下文

本项目主要围绕飞书、多维表格、知识星球、微信公众号、WaytoAGI、LLM 分类等外部服务做采集、解析、整理和同步。代码中存在较多外部 API、浏览器登录态、本机密钥配置、本地缓存和增量状态。

修改前应理解以下事实：

- `cfg/config.yaml` 保存业务配置，不应放敏感凭证。
- 敏感凭证通常来自本机密钥配置，例如 `~/.config/secrets/*.yaml`。
- `data/`、`log-err/`、缓存文件和状态文件可能包含运行产物，修改前要确认是否与当前任务直接相关。
- 外部服务失败不等于“无数据”，需要区分认证、权限、限流、网络、服务端异常和响应结构变化。

## 2. 工作前检查

开始修改前，先做最小必要调查：

1. 明确用户目标和成功标准。
2. 查看相关代码、配置和文档，不凭记忆修改。
3. 检查是否已有未提交变更，避免覆盖用户或其它 session 的工作。
4. 如果涉及外部访问、Token、权限、限流、重试或下载流程，必须先阅读：
   - `docs/technical/ACCESS_FAILURE_DIAGNOSTICS_PLAN.md`
   - 必要时同时阅读 `docs/technical/TOKEN_MANAGEMENT.md`
5. 如果涉及飞书、多维表格、URL 解析、日志缓存规范，优先查阅：
   - `docs/technical/ARCHITECTURE.md`
   - `docs/technical/CSV_LOG_SPEC.md`
   - `docs/technical/BITABLE_NOTES.md`

如果需求存在多种合理解释，先说明假设；如果假设会影响数据、凭证、外部写入或状态推进，应先向用户确认。

## 3. 变更原则

### 简单优先

- 只做当前任务需要的改动。
- 不为单一场景提前设计大型抽象。
- 不顺手重构无关代码。
- 能用现有项目模式解决时，不引入新框架或新依赖。

### 外科手术式修改

- 每一处改动都应能追溯到用户需求或验证要求。
- 不格式化无关文件。
- 不删除或回滚自己未创建的变更。
- 遇到脏工作区时，只处理当前任务相关文件。

### 可接力

- 复杂或渐进式工作要落到文档或台账。
- 本 session 形成的设计判断，如果会影响后续实现，必须写入项目文档。
- 完成阶段性工作后，更新对应文档中的状态、验证方式和剩余事项。

## 4. 开发流程

一般任务按以下流程推进：

1. 定义成功标准。
2. 阅读相关代码和文档。
3. 如为 bug 修复，先补能复现问题的测试。
4. 做最小实现。
5. 运行聚焦测试。
6. 在风险较高或影响范围较广时运行相关全量测试。
7. 更新必要文档或推进台账。
8. 向用户说明改了什么、如何验证、还有什么风险。

不要只停留在方案描述；除非用户明确只要分析，否则应把可执行的修复或文档变更落地。

## 5. 测试与验证

测试选择应匹配风险：

- 纯文档变更：检查链接、标题和引用路径即可。
- 小范围函数改动：运行对应单元测试。
- 影响外部访问、状态推进、下载或写入：补充失败场景测试，并运行相关测试集。
- 修改公共模块：新增公共模块测试，并运行所有依赖该模块的调用方测试。

常用测试命令应优先使用项目当前可用方式。若测试依赖 `src` 模块路径，使用：

```bash
PYTHONPATH=src python -m unittest discover -s tests -p 'test_*.py'
```

如果因为凭证、网络、沙箱或外部服务限制无法完整验证，必须在最终说明中明确：

- 哪些检查已经完成。
- 哪些检查未完成。
- 未完成的原因。
- 用户下一步可以如何验证。

## 6. 外部访问故障诊断

所有涉及 HTTP/API/下载/Token 的改动，都必须遵守：

- `docs/technical/ACCESS_FAILURE_DIAGNOSTICS_PLAN.md`

关键要求：

- 不把最终访问失败误报为“无数据”。
- 如果某次尝试失败但最终重试成功，终端应明确说明“重试成功”。
- 认证失败、权限失败、限流、网络失败、服务端错误应尽量给出不同提示。
- `401`、`403` 等通常不应盲目多次重试。
- 只有完整访问成功后，才允许把结果当作正常空结果处理。
- 状态推进字段，例如 `last_download_date`，只有在数据完整性可确认时才更新。

如果新增或改造访问故障判断，应同步更新 `docs/technical/ACCESS_FAILURE_DIAGNOSTICS_PLAN.md` 的“推进台账”。

## 7. 配置、凭证与本地状态

### 配置

- 业务配置放在 `cfg/config.yaml`。
- 不要把本机绝对路径改回 Windows 路径或其它机器路径，除非用户明确要求。
- 修改目录配置前，应确认目标系统和实际挂载路径。

### 凭证

- 不要把真实 token、cookie、secret 写入仓库。
- 不要新增 `credentials.yaml` 等敏感文件到仓库。
- 如果需要提示用户修复凭证，应指出配置来源和键名，但不要打印完整密钥值。

### 状态文件

- 增量状态只在任务确认成功后推进。
- 如果抓取中断、认证失败、权限失败或响应不完整，不应更新最后处理时间。
- 修改状态字段时，应说明对应的人类可读时间和原始时间戳。

## 8. 终端回显规范

终端输出应帮助用户判断“是否正常、哪里失败、下一步做什么”。

建议输出包含：

- 当前处理对象，例如群组、文档、URL、文件。
- 时间范围或处理范围。
- 目标目录或写入位置。
- 重试过程和最终结果。
- 状态是否更新及原因。

避免：

- 用“获取失败，停止”后又输出“无文件需要下载”造成误导。
- 只打印底层异常，不说明用户可采取的行动。
- 输出大量无结构日志淹没有用信息。

## 9. 文档与台账

本项目已经有文档分区：

- `docs/guides/`：面向用户的使用说明。
- `docs/technical/`：面向开发者的技术说明和工程规范。
- `docs/prd/`：产品需求文档。
- `docs/archive/`：历史归档。

新增或修改文档时：

- 技术规范优先放入 `docs/technical/`。
- 用户操作说明优先放入 `docs/guides/`。
- 新增重要文档后，同步更新 `docs/README.md`。
- 渐进式工作必须有明确台账，记录状态、范围、验证方式和备注。

## 10. 代码风格

- 优先匹配现有代码风格。
- 使用清晰函数名和变量名，不用晦涩缩写。
- 只在复杂逻辑前添加有价值的短注释。
- 避免大段无必要注释或重复解释代码本身。
- 解析结构化数据时，优先使用结构化 API 或标准解析器，不做脆弱字符串拼接。
- 新增公共模块前，确认至少有两个明确调用场景，或已有台账说明将分阶段接入。

## 11. Git 与文件安全

- 不执行破坏性命令，除非用户明确要求。
- 不回滚用户或其它 session 的改动。
- 不把运行产物、缓存、日志批量加入版本控制，除非任务明确要求。
- 提交前应检查 diff，确保只包含当前任务相关文件。

## 12. 完成标准

一次任务完成时，应满足：

- 用户提出的问题已被回答或变更已落地。
- 相关测试或核查已执行。
- 未验证部分已明确说明。
- 如果是渐进式工作，台账已更新。
- 最终回复包含关键文件、验证方式和剩余风险。
